# Code Style Rules — MarketUK

## General

- TypeScript strict mode is on — no `any` unless absolutely necessary (admin mock data is an exception)
- No comments unless the WHY is non-obvious (a workaround, hidden constraint, subtle invariant)
- No docstrings or multi-line comment blocks
- Do not add unused imports, variables, or dead code
- Prefer editing existing files over creating new ones

## React

### Hooks Rule — CRITICAL
All `useState`, `useEffect`, `useRef`, `useMemo`, `useCallback` calls MUST appear before any conditional return. React crashes the entire app if hooks are called conditionally.

```tsx
// ✅ CORRECT
function Component() {
  const { user, loading } = useAuth();
  const [tab, setTab] = useState("overview"); // hook before return

  if (loading || !user) return null; // early return after hooks
}

// ❌ WRONG — will crash
function Component() {
  const { user, loading } = useAuth();
  if (loading || !user) return null;
  const [tab, setTab] = useState("overview"); // hook after return
}
```

### Component patterns
- Prefer function declarations over arrow functions for named components
- Keep components focused — split large components into smaller sub-components in the same file
- Use `React.ReactNode` for children prop type

### State management
- Local `useState` only — no global state library
- For shared auth state use `useAuth()` from `@/lib/auth`
- Lift state only when 2+ siblings need it

## File & Import Conventions

- Path alias: `@/` maps to `src/` — always use `@/` for absolute imports
- shadcn/ui components live in `src/components/ui/` — import from `@/components/ui/`
- Site-specific components live in `src/components/site/`
- Never edit files in `src/components/ui/` — they are generated by shadcn

```tsx
import { Button } from "@/components/ui/button";
import { ListingCard } from "@/components/site/ListingCard";
import { useAuth } from "@/lib/auth";
import { supabase } from "@/lib/supabase";
import { toast } from "sonner";
```

## Routing (TanStack Router)

- All routes are file-based in `src/routes/`
- Nested routes use dot notation: `dashboard.buyer.tsx` → `/dashboard/buyer`
- Dynamic segments use `$`: `listings.$id.tsx` → `/listings/:id`
- Always use `createFileRoute` for the route export

```tsx
export const Route = createFileRoute("/listings/$id")({
  component: ListingDetail,
});
```

- Navigation: use `useNavigate()` hook or `<Link to="/path" />`
- Never use `window.location.href` for internal navigation

## Styling

- Tailwind utility classes only — no inline `style={{}}` except for dynamic values that can't be expressed as Tailwind
- Brand colours: `text-gold`, `bg-gold` (gold accent `#C8922A`), `text-primary` (navy `#0D2B4E`)
- Admin panel uses hardcoded hex values: `bg-[#1C2B3A]`, `border-[#2a3d52]`, `text-[#C8922A]`
- Responsive: mobile-first — `sm:`, `md:`, `lg:` breakpoints
- Use `cn()` from `@/lib/utils` to merge conditional class names

```tsx
import { cn } from "@/lib/utils";
<div className={cn("base-class", isActive && "active-class")} />
```

## Error Handling & Feedback

- Always use `toast` from sonner for user-facing feedback — never `alert()` or `console.log` in production
- Handle both success and error cases for every async operation
- Show loading states for async data (spinner, disabled button, or skeleton)

```tsx
const [saving, setSaving] = useState(false);

async function handleSubmit() {
  setSaving(true);
  const { error } = await supabase.from("listings").insert(payload);
  if (error) { toast.error("Failed to save listing."); setSaving(false); return; }
  toast.success("Listing created!");
  setSaving(false);
}
```

## Forms

- Use React Hook Form + Zod for complex forms
- For simple admin forms, controlled `useState` is fine
- Always validate required fields before calling Supabase
- Show inline error messages (not just toasts) for field-level errors

## TypeScript

- Define types at the top of the file or in a dedicated `types.ts`
- Use `type` for object shapes, `interface` sparingly
- Supabase generated types are in `@/lib/database.types.ts`
